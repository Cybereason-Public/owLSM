# If I remember correctly the full flow is:
## RuleGenerator:
1. take rule condition
2. expand parenthesis and words like "and", "or", "not"
3. convert to AST 
4. walk AST and create the id_to_string, id_to_ip and id_to_predicate tables. 
5. convert to postfix
6. convert to tokens
7. sirealize as json

## Userspace (C++ code)
8. Deserialze the json into structs.
    + for ip's we had to convert it to numeric representation
    + for strings we needed to do: length, is contains and if yes create DFA

9. Populate the bpf maps.
    + rule maps: so each event type will only be checked by its relevant rules. 
    + string map, dfa map, ip map, etc'

## In kernel: After populating event struct
10. iterate over the rules of the event type. using bpf_for_each_map_elem
11. push rule tokens into self implemented stack
12. Start a two stack evaluation of the tokens (representing the postfix rule). 
    - For each token in the rule, check if its a predicate and it yes:
        + Check if we already proccessed it against the current event, using the predicate cache
        + Do the comparison operation (string compare, ip compare or numeric)
        + save the result in the cache
13. Based on the two stack evaluation, determine if the event matches the rule. If yes, return the rule id and its action.
